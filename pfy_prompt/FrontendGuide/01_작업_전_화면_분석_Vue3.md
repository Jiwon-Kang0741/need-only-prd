# 01. 작업 전 화면 분석 (Vue3 개발·수정용)

## 📋 이 단계의 목적

- **수정 시**: 변경 대상 페이지·컴포넌트와 그 영향 범위(API, 타입, 스토어, 라우트)를 빠르게 파악합니다.
- **신규 시**: 비슷한 기존 화면을 참고해 구조·패턴·API 설계 방향을 정합니다.
- **공통**: 작업 전에 현재 Vue3 코드베이스 구조와 데이터 흐름을 이해하여, `MigFrontend.mdc` 및 `AI_AGENT_DEVELOPMENT_GUIDE.md`와 맞춰 일관되게 개발·수정할 수 있도록 합니다.

---

## 🔍 분석 대상

| 대상 | 위치 / 확인 방법 | 목적 |
|------|------------------|------|
| **페이지 진입점** | `src/pages/[module]/[category]/[screenId]/index.vue` | 레이아웃, provide/inject, 사용 컴포넌트 파악 |
| **화면 전용 컴포넌트** | `src/pages/.../components/` (SearchForm, DataTable, SumGrid 등) | 역할, props, emit, inject 관계 파악 |
| **API·타입** | `src/api/pages/[module]/[category]/[screenId].ts`, `types.ts` | 엔드포인트, 요청/응답 타입, 기존 호출 패턴 확인 |
| **라우트** | `src/router/` 또는 동적 라우트 등록 위치 | URL, 메뉴 ID, 레이아웃 연결 확인 |
| **스토어·공통코드** | `src/stores/` (예: commonCodeStore) | 검색 조건 옵션, 공통 상태 사용처 파악 |
| **공통 컴포넌트** | `@/components/common/...` import 목록 | 프로젝트 컨벤션 및 재사용 컴포넌트 확인 |

필요 시 백엔드(Service/DAO, Mapper XML)는 **참고용**으로만 확인합니다. 상세는 `06_API_백엔드_연동.md` 등을 참고하세요.

## 📝 Step 1: 대상 페이지 구조 파악

### 1.1 index.vue 확인

- **ContentHeader**: `menuId`(화면 ID) 연결 여부
- **레이아웃 블록**: 검색 영역 / 집계(SumGrid) / 테이블(DataTable) 등 구역 구분
- **provide / inject**: 예) `searchParams` 제공·주입 여부 및 타입
- **하위 컴포넌트**: 어떤 화면 전용 컴포넌트를 쓰는지, 데이터는 props인지 inject인지

```text
예시 구조:
index.vue
  ├── ContentHeader
  ├── [ScreenId]SearchForm   (검색 조건)
  ├── [ScreenId]SumGrid      (집계, 있을 경우)
  ├── [ScreenId]DataTable    (목록)
  └── [ScreenId]ProgressList (요청 시에만)
```

### 1.2 폴더·파일 구조 확인

- **필수**: `index.vue`, `[screenId].scss`
- **components/**: `[screenId]SearchForm`, `[screenId]DataTable`, `[screenId]SumGrid` 등
- **DataTable만** `utils/index.ts` 사용 (컬럼 정의, getRows 등). SumGrid·SearchForm에는 utils 폴더 없음.

### 1.3 타입·API 위치 (CRITICAL)

- **타입**: `src/api/pages/[module]/[category]/types.ts` 에 정의 (화면 폴더 내부 `types.ts` 아님)
- **API 함수**: `src/api/pages/[module]/[category]/[screenId].ts`
- **import**
  - API 파일: `from './types'`
  - 페이지/컴포넌트: `from '@/api/pages/[module]/[category]/types'`

이 화면에서 사용하는 API 함수 목록과, 응답이 단일 객체인지 배열(List)인지 확인해 두면 수정·신규 설계에 유리합니다.

## 📝 Step 2: 데이터·상태 흐름 파악

### 2.1 검색 조건 (searchParams)

- **정의·제공**: 보통 `index.vue`에서 `ref`로 정의 후 `provide('searchParams', searchParams)`
- **SearchForm**: watch로 `searchParams` 변경을 구독 → 폼 필드 동기화 (`setFieldValue` 등). 조회 시 `searchParams` 직접 갱신 후 API 호출
- **조건**: 빈 문자열도 유효할 수 있으면 `if (newVal !== undefined)` 사용 (단순 `if (newVal)` 지양)

### 2.2 목록·집계 데이터

- **ref 이름**: 예) `fetchedMainData`, `fetchedSumData`
- **API 호출 시점**: 초기 로드, 조회 버튼, SumGrid/ProgressList 버튼 클릭 등
- **응답 형태**: 단일 객체 vs 배열에 따라 할당 방식 구분 (예: `fetchedSumData.value = response` vs `response[0]`)

### 2.3 공통코드

- **스토어**: `useCommonCodeStore()` → `loadMulti`, `options`
- **로딩 시점**: `onMounted` 또는 특정 필드 변경 시(동적 로딩)
- **옵션 사용**: SelectBox 등에서 `commonCodeStore.options(QueryId, ...)` 호출 시, `loadMulti`에 넘긴 `customParam`과 `options()` 인자가 일치해야 함

### 2.4 이벤트 흐름

- **조회**: SearchForm emit 또는 상위에서 `searchParams` 반영 후 API 호출
- **저장/삭제 등**: 버튼 → 확인(ConfirmDialog) → API → Toast
- **행 선택**: DataTable 선택 시 하이라이트, 상세 오픈 등 후속 처리 여부

작업 시 "이 화면의 API 호출 목록과 트리거"를 한 표로 정리해 두면 수정 범위를 잡기 쉽습니다.

## 📝 Step 3: 참고할 기존 화면 찾기

### 3.1 검색 방법

- **같은 모듈/카테고리**: `src/pages/[module]/[category]/` 내 비슷한 화면
- **규칙/가이드에서 추천**: `MigFrontend.mdc`, `AI_AGENT_DEVELOPMENT_GUIDE.md` 에서 언급된 참조 화면 (예: spov010, pmdp020)
- **PFY(pfy-front) 설계 참고 화면**: `src/pages/sy/ds/` 하위 (예: `src/pages/sy/ds/pmdp020`, `src/pages/sy/ds/spov010`, `src/pages/sy/ds/spod010`)
- **grep 활용**:
  - 공통코드 QueryId: `grep -r "MNPS010Query" src/pages --include="*.vue"`
  - API 경로: `grep -r "api/v1/SPOV010" src --include="*.ts"`
  - 컴포넌트 사용: `grep -r "SearchForm" src/pages --include="*.vue"`

### 3.2 참고할 내용

| 항목 | 참고 포인트 |
|------|-------------|
| **SearchForm** | watch·inject 패턴, `setFieldValue` / `getValues`, searchParams 갱신 방식 |
| **API** | axios 사용, `formatErrorMessage`, `response.data.payload` 반환, 타입 import 경로 |
| **SumGrid / DataTable** | ref 이름, API 연동 시점, DataTable의 경우 `utils/index.ts` (getColumns, getRows) |
| **스타일** | SCSS 구조, `:deep()`, CSS 변수, 레이아웃( flex/grid ) |

## 📝 Step 4: 수정/신규별 작업 범위 정리

### 4.1 수정 시

- **변경 파일 목록**: 수정할 `.vue`, `.ts`, `.scss` 나열
- **영향 범위**: 타입 추가/변경, API 시그니처 변경, 스토어·라우트 변경 여부
- **추가/삭제**: 새 API 엔드포인트, 공통코드 QueryId, 컴포넌트 추가/제거

### 4.2 신규 시

- **페이지 경로**: `src/pages/[module]/[category]/[screenId]/`
- **API·타입**: `src/api/pages/[module]/[category]/[screenId].ts`, `types.ts` (같은 카테고리 `types.ts`에 타입 추가)
- **생성할 컴포넌트**: SearchForm, DataTable, SumGrid 등. ProgressList는 요청 시에만 추가
- **참고 화면**: 비슷한 기존 화면 1~2개 지정 (임포트 패턴, watch, API 패턴 복사)

```markdown
# [화면ID] 작업 전 분석

## 개요
- **화면 ID**: (예: SPOV010)
- **작업 유형**: 수정 / 신규
- **참고 화면**: (예: `sy/ds/pmdp020`, `sy/ds/spov010` — PFY 기준 경로)

## 페이지 구조
- index.vue: ContentHeader, SearchForm, SumGrid, DataTable
- provide: searchParams

## API 목록
| 용도     | 함수명                    | 요청 타입 | 응답 형태   |
|----------|---------------------------|-----------|-------------|
| 집계     | fetchSelectOverviewSummary | SearchParams | 단일 객체 |
| 목록     | fetchSelectOverviewList    | SearchParams | 배열     |

## 데이터 흐름
- searchParams: index에서 provide → SearchForm에서 watch로 폼 동기화, 조회 시 갱신 후 API 호출
- fetchedSumData, fetchedMainData: API 응답 할당

```
