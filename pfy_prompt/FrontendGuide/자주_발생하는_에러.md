# 자주 발생하는 에러 및 해결 방법

## 📋 이 문서의 목적

PMDP010 개발 과정에서 실제로 발생한 에러와 해결 방법을 정리합니다.

---

## 1️⃣ 성능 관련 에러

### 1.1 DataTable 스크롤이 매우 느림

**증상**:
- 대용량 데이터 (1000개 이상) 조회 시 로딩 3초 이상
- 스크롤 시 버벅임 (15 FPS 이하)
- 브라우저 탭이 응답 없음

**원인**:
```vue
<!-- ❌ WRONG -->
<DataTable
  :value="fetchedMainData"
  scrollHeight="flex"
>
```

**문제점**:
- `scrollHeight="flex"`: 모든 행을 DOM에 렌더링
- 1000개 행 × 27개 컬럼 = 27,000개 DOM 노드 생성

**해결**:
```vue
<!-- ✅ CORRECT -->
<DataTable
  :value="fetchedMainData"
  scrollHeight="540px"
  :virtualScrollerOptions="{ itemSize: 46 }"
>
```

**결과**:
- DOM 노드: 27,000개 → ~300개
- 로딩 시간: ~3초 → ~100ms
- 스크롤 FPS: ~15 → ~60

---

### 1.2 행 클릭/hover 시 극도로 느림

**증상**:
- 행 클릭 시 0.5초 이상 지연
- hover 시 깜빡임
- Chrome DevTools에서 함수 호출 5000번+

**원인**:
```vue
<!-- ❌ WRONG - 템플릿 슬롯 포맷팅 -->
<Column field="ASK_DT" header="의뢰일자">
  <template #body="{ data }">
    {{ formatDate(data.ASK_DT, 'YYYYMMDD', 'YYYY-MM-DD') }}
  </template>
</Column>
```

**문제점**:
- 렌더링 시마다 `formatDate` 함수 호출
- 5개 날짜 컬럼 × 1000개 행 = 5000번 호출/렌더링

**해결**:
```typescript
// ✅ CORRECT - watch에서 한 번만 포맷팅
watch(
  () => props.fetchedMainData,
  async (newData) => {
    if (!newData) return;
    await nextTick();
    
    newData.forEach((row: any) => {
      if (row.ASK_DT) {
        row.ASK_DT_FORMATTED = formatDate(row.ASK_DT, 'YYYYMMDD', 'YYYY-MM-DD');
      }
    });
  },
  { immediate: true }
);
```

```vue
<!-- 포맷팅된 필드 사용 -->
<Column field="ASK_DT_FORMATTED" header="의뢰일자" />
```

**결과**:
- 함수 호출: 5000번+ → 1000번 (데이터 로딩 시 1회)
- 행 클릭 지연: 0.5초 → 즉시

---

### 1.3 행 hover 시 전체 테이블 리플로우

**증상**:
- hover 시 모든 행이 깜빡임
- FPS 드롭 (60 → 20)

**원인**:
```scss
/* ❌ WRONG - GPU 가속 없음 */
.p-datatable-tbody tr:hover {
  background-color: var(--primary-50);
}
```

**해결**:
```scss
/* ✅ CORRECT - GPU 가속 */
.p-datatable-tbody tr {
  will-change: background-color;
  
  &:hover:not(.p-datatable-row-selected) {
    background-color: var(--primary-50);
  }
}

.p-datatable-tbody > tr > td {
  contain: layout style;
}
```

**결과**:
- hover FPS: 20 → 60
- 리플로우: 전체 테이블 → 해당 행만

## 2️⃣ 상태 관리 에러

### 2.1 진행상태 버튼 클릭 시 SelectBox가 업데이트되지 않음

**증상**:
- SumGrid 버튼 클릭 → SelectBox 값 변하지 않음
- 조회는 정상 작동
- 콘솔 에러 없음

**원인 (watch 조건)**:
```typescript
// ❌ WRONG - null과 빈 문자열을 놓침
watch(
  () => searchParams.value?.prg_stat,
  (newVal) => {
    if (newVal) {  // ← 문제!
      searchFormRef.value.form.setFieldValue('prg_stat', newVal);
    }
  }
);
```

**문제점**:
- `if (newVal)`: falsy 값 (null, '', 0) 모두 제외
- 전체 버튼 클릭 (`prg_stat = null`) → 조건 false → SelectBox 업데이트 안 됨

**해결**:
```typescript
// ✅ CORRECT - undefined만 제외
watch(
  () => searchParams.value?.prg_stat,
  (newVal) => {
    if (newVal !== undefined && searchFormRef.value?.form) {
      searchFormRef.value.form.setFieldValue('prg_stat', newVal ?? '');
    }
  }
);
```

**이유**:
| 값 | `if (newVal)` | `if (newVal !== undefined)` |
|----|--------------|----------------------------|
| `'S01'` | ✅ true | ✅ true |
| `null` | ❌ false | ✅ true |
| `''` | ❌ false | ✅ true |
| `undefined` | ❌ false | ❌ false |

### 2.2 searchFormRef.value.form is undefined

**증상**:
```
TypeError: Cannot read properties of undefined (reading 'setFieldValue')
```

**원인 (Ref 접근 경로)**:
```typescript
// ❌ WRONG - .value.value
searchFormRef.value.value.form.setFieldValue('prg_stat', newVal);

// ❌ WRONG - .form.value
searchFormRef.value.form.value.setFieldValue('prg_stat', newVal);
```

**해결**:
```typescript
// ✅ CORRECT - .value.form
searchFormRef.value.form.setFieldValue('prg_stat', newVal ?? '');
```

**이유**:
```typescript
// SearchForm.vue
const searchFormRef = ref();
defineExpose({ searchFormRef });  // ← 이미 ref를 expose

// Parent.vue
const searchFormRef = ref();  // ← 이것도 ref

// Vue의 자동 unwrap
searchFormRef.value               // → expose된 객체
searchFormRef.value.searchFormRef // → 내부 ref
searchFormRef.value.form          // → 실제 form 인스턴스
```

### 2.3 복잡한 ref 체인 사용으로 인한 에러

**증상**:
```

```

**원인 (과거 패턴)**:

- 백엔드 HQML: `PRG_STAT`이 null
- SQL 조건이 적용되지 않음

**원인**:

**HQML 기대값**:
```xml

</#if>
```

**해결**:
```typescript
// ✅ CORRECT - convertSearchParamsToUpperCase 사용
function convertSearchParamsToUpperCase(params: any): any {
  const fieldMapping = {
    prg_stat: 'PRG_STAT',
    cont_no: 'CONT_NO',
    // ...
  };
  
  const converted: any = {};
  
  Object.keys(params).forEach((key) => {
    const value = params[key];
    const upperKey = fieldMapping[key] || key.toUpperCase();
    
    if (value !== null && value !== undefined && value !== '') {
      converted[upperKey] = value;
    }
  });
  
  return converted;
}

// 사용
const convertedParams = convertSearchParamsToUpperCase(params);
const response = await api.post<ApiResponse<any>>('/api/v1/PMDP010-selectDpAskList', convertedParams);
// ← { PRG_STAT: 'S01', CONT_NO: '1234' } 대문자로 변환됨
```

```typescript
// ✅ CORRECT - YYYYMMDD 문자열로 변환
function formatDateToYYYYMMDD(date: Date): string {
  const year = date.getFullYear();
  const month = String(date.getMonth() + 1).padStart(2, '0');
  const day = String(date.getDate()).padStart(2, '0');
  return `${year}${month}${day}`;
}

// 사용
if (params.searchDt && Array.isArray(params.searchDt)) {
  const [startDate, endDate] = params.searchDt;
  
  if (startDate) {
    converted.SEARCHDT_FROM = formatDateToYYYYMMDD(startDate);  // '20240115'
  }
  
  if (endDate) {
    converted.SEARCHDT_TO = formatDateToYYYYMMDD(endDate);
  }
}
```

## 6️⃣ UI/UX 관련 에러

### 6.1 alert() 사용으로 인한 UX 저하

**증상**:
- 브라우저 기본 alert 창이 표시됨
- 디자인이 tomms-lite-front와 맞지 않음
- 모달이 차단되어 페이지 전체가 블로킹됨

**원인**:
```typescript
// ❌ WRONG - alert() 사용
const handleSave = () => {
  if (!formData.value.name) {
    alert('이름을 입력하세요.');
    return;
  }
  
  if (confirm('저장하시겠습니까?')) {
    saveData();
  }
};
```

**문제점**:
- `alert()`, `confirm()`, `prompt()`: 브라우저 기본 다이얼로그
- UX 저하: 모든 작업이 블로킹됨
- 디자인 불일치: tomms-lite-front 테마와 다름
- 모바일 비호환: 작은 화면에서 문제

**해결**:
```typescript
// ✅ CORRECT - Toast 사용
import { useToast } from 'primevue/usetoast';

const toast = useToast();

const handleSave = () => {
  // 검증 메시지
  if (!formData.value.name) {
    toast.add({
      severity: 'warn',
      summary: '경고',
      detail: '이름을 입력하세요.',
      life: 3000
    });
    return;
  }
  
  // 확인이 필요한 경우 ConfirmDialog 사용
  confirm.require({
    message: '저장하시겠습니까?',
    acceptLabel: '저장',
    rejectLabel: '취소',
    accept: () => {
      saveData();
    }
  });
};

// 성공/실패 피드백
const saveData = async () => {
  try {
    await api.post('/save', formData.value);
    
    toast.add({
      severity: 'success',
      summary: '저장 완료',
      detail: '데이터가 저장되었습니다.',
      life: 3000
    });
  } catch (error) {
    toast.add({
      severity: 'error',
      summary: '저장 실패',
      detail: '저장 중 오류가 발생했습니다.',
      life: 3000
    });
  }
};
```

**Toast vs ConfirmDialog 선택 기준**:
| 상황 | 사용 컴포넌트 | 이유 |
|------|------------|------|
| 검증 실패 | Toast (warn) | 단순 알림, 사용자 액션 불필요 |
| 저장 성공 | Toast (success) | 피드백, 자동 닫힘 |
| API 오류 | Toast (error) | 오류 알림 |
| 삭제 확인 | ConfirmDialog | 사용자 확인 필요, 위험한 액션 |
| 저장 확인 | ConfirmDialog | 사용자 확인 필요 |
| 선택 안내 | Toast (info) | 정보성 메시지 |

**규칙**:
- ❌ **절대 금지**: `alert()`, `confirm()`, `prompt()`
- ✅ **단순 알림**: Toast 사용
- ✅ **확인 필요**: ConfirmDialog 사용

## 3.0 백엔드 응답 구조 불일치

### 3.0.1 데이터가 렌더링되지 않음 (List 반환 패턴)

**증상**:
- 백엔드에서 데이터는 정상 조회됨
- 프론트엔드에서 `fetchedSumData.value.dsOutput[0]` 접근 시 `undefined`
- 화면에 데이터가 표시되지 않음

**원인**:
```typescript
// ❌ WRONG - List 반환인데 dsOutput 구조를 기대
const response = await selectDpProgressSum(requestParams);
if (response.header.responseCode === 'S0000') {
  fetchedSumData.value = response.payload;  // payload가 List
}

// SumGrid에서 접근
const sumData = computed(() => {
  if (fetchedSumData.value && fetchedSumData.value.dsOutput) {  // ← dsOutput 없음!
    return fetchedSumData.value.dsOutput[0];
  }
});
```

**백엔드 응답 구조**:
```java
// PMDP100ServiceImpl.java
@Override
public List<Pm100SelectProgressSumResDto> selectDpProgressSum(...) {
    return pmdp100Dao.selectDpProgressSum(request);  // List 직접 반환
}
```

**문제점**:
- 백엔드는 `List<Pm100SelectProgressSumResDto>`를 직접 반환
- 프론트엔드는 `{ dsOutput: [...] }` 형태를 기대
- `response.data.payload`가 List이므로 `payload[0]` 또는 직접 사용해야 함

**해결**:
```typescript
// ✅ CORRECT - List를 직접 반환하도록 API 수정
export const selectDpProgressSum = async (
  params: SearchParams
): Promise<SumData[]> {  // ← List 타입
  const response = await api.post<ApiResponse<SumData[]>>(
    API_ENDPOINTS.selectDpProgressSum,
    params
  );
  
  return response.data.payload;  // ← List 직접 반환
}

// 화면 컴포넌트에서 사용
const response = await selectDpProgressSum(requestParams);
// 집계 데이터는 첫 번째 항목 사용
fetchedSumData.value = response && response.length > 0 ? response[0] : null;
// 목록 데이터는 전체 List 사용
fetchedMainData.value = response || [];

// SumGrid에서 접근
const sumData = computed(() => {
  if (fetchedSumData.value) {  // ← 직접 접근
    return fetchedSumData.value;
  }
  return { sumtot: 0, suma: 0, /* ... */ };
});
```

**판단 기준**:
1. 백엔드 서비스 메서드 반환 타입 확인
   - `Pm020SelectStatSumResDto` → 단일 객체 → `response.data.payload` 직접 사용
   - `List<Pm100SelectProgressSumResDto>` → List → `response.data.payload[0]` 또는 직접 사용
2. 프론트엔드 타입 정의
   - 단일 객체: `PMDP020SelectStatSumResponse`
   - List: `PMDP100SumData[]`
3. 화면 컴포넌트에서 데이터 접근
   - 단일 객체: `fetchedSumData.value` 직접 사용
   - List (집계): `fetchedSumData.value[0]` 또는 첫 번째 항목
   - List (목록): `fetchedMainData.value` 전체 배열

**DataTable에서 변환 필요 시**:
```typescript
// utils/index.ts
function toUpperSnakeCase(str: string): string {
  return str.replace(/([A-Z])/g, '_$1').toUpperCase();
}

function convertKeysToUpperSnakeCase(obj: any): any {
  if (!obj || typeof obj !== 'object') {
    return obj;
  }

  const result: any = {};
  for (const [key, value] of Object.entries(obj)) {
    const upperKey = toUpperSnakeCase(key);
    result[upperKey] = value;
  }
  return result;
}

export function getRows(data: any[]): any[] {
  return data.map((row, index) => {
    const convertedRow = convertKeysToUpperSnakeCase(row);  // ← 변환
    return {
      ...convertedRow,
      rowIndex: index + 1,
    };
  });
}
```

**판단 기준**:
1. 백엔드 DTO 필드명 확인
   - `Pm100SelectProgressSumResDto.java`에서 필드명 확인
   - camelCase면 프론트엔드도 camelCase로 정의
2. DataTable 컬럼 정의 확인
   - `getColumns()`에서 `field: 'CONT_NO'` (대문자) 사용
   - `getRows()`에서 변환 필요
3. SumGrid/ProgressList에서 필드 접근
   - camelCase로 정의했으면 소문자로 접근
   - 또는 변환 함수 사용
