# 00. 시작하기

## 📋 이 문서의 목적

이 가이드는 **Vue 3 + TypeScript (tomms-lite-front)** 프로젝트에서 **신규 화면 개발**과 **기존 화면 수정**을 할 때 따를 **공통 절차와 패턴**을 제공합니다.

- **신규 개발**: 요구사항과 유사 기존 화면을 참고해 `pages`·`api` 구조에 맞춰 구현합니다.
- **수정 개발**: 변경 대상 페이지·컴포넌트·API·타입의 영향 범위를 파악한 뒤 수정합니다.
- **공통**: `AI_AGENT_DEVELOPMENT_GUIDE.md` 및 본 FrontendGuide의 패턴을 준수합니다.

## 🎯 핵심 목표

1. **신규**: 기존 화면(예: PMDP020, spov010)과 동일한 폴더·컴포넌트·API 패턴으로 구현
   - **PFY(tomms-front)**: 패턴 참고용 샘플은 `src/pages/sy/ds/` 하위(예: `pmdp020`, `spov010`)에 둡니다.
2. **수정**: 기존 코드 스타일·패턴을 유지하며 변경
3. **스타일**: 기존 화면과 동일한 톤앤매너·스타일 가이드 준수
4. **성능 최적화 필수 적용** · **유지보수 가능한 코드 작성**

## 🏗️ 프로젝트 구조 이해

### 1. 프로젝트 위치

모든 Vue 개발은 `tomms-lite-front` 프로젝트에서 진행합니다.

```
tomms-lite-front/            # 🎯 개발 프로젝트
├── src/
│   ├── api/                 # API 레이어 (api/pages/[module]/[category]/)
│   ├── pages/               # 화면 (pages/[module]/[category]/[screenId]/)
│   ├── components/common/   # 공통 컴포넌트
│   └── stores/              # Pinia 스토어
│
tomms-lite-war/              # Spring Boot 백엔드 (API 연동 시 참고)
```

**개발 흐름**:
- **신규**: 요구사항 정리 → 유사 기존 화면 참고 → `pages/[module]/[category]/[screenId]` 및 `api/pages/...` 구현 → 라우트·API 연동
- **수정**: 변경 대상 `index.vue`·컴포넌트·API·타입·라우트 파악 → 수정 범위 정리 → 구현

### 2. tomms-lite-front 폴더 구조

신규 화면은 아래 구조로 추가하고, 수정 시에는 해당 화면 폴더만 변경합니다.

```
tomms-lite-front/src/
├── api/                 # API 통신 레이어
│   └── pages/[module]/[category]/
│       ├── [screenId].ts # API 함수들
│       └── types.ts     # TypeScript 타입 정의 (화면 폴더가 아님!)
│
├── pages/               # 화면 컴포넌트
│   └── [module]/[category]/[screenId]/
│       ├── index.vue    # 메인 페이지
│       ├── [screenId].scss # 페이지 스타일
│       └── components/  # 하위 컴포넌트들
│           ├── [screenId]SearchForm/
│           │   ├── [ScreenId]SearchForm.vue
│           │   └── [ScreenId]SearchForm.scss
│           ├── [screenId]SumGrid/      # 필요 시
│           │   ├── [ScreenId]SumGrid.vue
│           │   └── [ScreenId]SumGrid.scss
│           └── [screenId]DataTable/
│               ├── [ScreenId]DataTable.vue
│               ├── [ScreenId]DataTable.scss
│               └── utils/               # DataTable만 utils!
│                   └── index.ts        # getColumns, getRows 등
│
├── components/          # 공통 컴포넌트
│   └── common/
│       ├── searchForm/
│       ├── dataTable2/
│       ├── button/
│       └── ...
│
└── stores/             # 상태 관리
    ├── commonCodeStore.ts
    └── userStore.ts
```

## 🔑 핵심 개념

### 1. tomms-lite-front 패턴 준수

**절대 원칙**: tomms-lite-front에 이미 구현된 패턴이 있다면 **무조건 그대로 사용**

```typescript
// ❌ 잘못된 방법 - 직접 구현
const handleProgressClick = (field: string) => {
  // SearchForm의 setValue를 복잡하게 호출
  searchFormRef.value?.searchFormRef?.value?.form?.value?.setFieldValue(...)
}

// ✅ 올바른 방법 - 기존 패턴 그대로 (예: spov010)
const handleProgressClick = (field: string) => {
  // 1. searchParams만 업데이트
  searchParams.value.prg_stat = field;

  // 2. SearchForm의 watch가 자동으로 SelectBox 업데이트
  // (SearchForm 내부에서 watch 구현)
}
```

### 2. searchParams 중심 아키텍처

```typescript
// index.vue (부모)
const searchParams = ref();
provide('searchParams', searchParams);

// SearchForm (자식)
const searchParams = inject('searchParams');
watch(() => searchParams.value?.prg_stat, (newVal) => {
  if (newVal !== undefined && searchFormRef.value?.form) {
    searchFormRef.value.form.setFieldValue('prg_stat', newVal ?? '');
  }
});

// ProgressBar (형제)
const searchParams = inject('searchParams');
const handleClick = () => {
  searchParams.value = { ...searchParams.value, prg_stat: 'S01' };
};
```

**핵심**: `searchParams`를 변경하면 → `watch`가 감지 → 폼 필드 자동 동기화

### 3. API 파라미터 규약

기존 백엔드(tomms-lite-war)가 **HQML·대문자 파라미터**를 사용하는 경우, API 레이어에서 변환합니다. 신규 백엔드가 소문자만 사용하면 변환 불필요합니다.

```typescript
// 백엔드가 대문자 파라미터를 기대하는 경우
// Vue에서는 소문자로 작성
searchParams.value = { prg_stat: 'S01' };

// API 레이어에서 변환 (06_API_백엔드_연동.md 참고)
// convertToUpperCase(params) 등
```

### 4. Ref 접근 경로 (매우 중요!)

```typescript
// ❌ 잘못된 경로
searchFormRef.value?.form?.value.setFieldValue()

// ✅ 올바른 경로 (Vue의 ref auto-unwrap)
searchFormRef.value?.form.setFieldValue()

// 구조 설명:
// - searchFormRef.value → 화면 SearchForm 컴포넌트 인스턴스
// - .form → SearchForm이 expose한 ref (자동 unwrap됨!)
// - .setFieldValue() → PrimeVue Form 메서드
```

## ⚠️ 시작 전 주의사항

### DO (반드시 해야 할 것)

1. ✅ **tomms-lite-front를 먼저 확인**
   - 유사한 화면이 있는지 찾기
   - 패턴과 스타일 참고

2. ✅ **신규 시: 유사 기존 화면 참고**
   - 같은 모듈/카테고리 내 비슷한 화면의 폴더 구조·SearchForm/DataTable/API 패턴 확인
   - 참고 화면 1~2개를 정한 뒤 그 구조를 따르기

3. ✅ **수정 시: 영향 범위 파악**
   - 변경 대상 페이지·컴포넌트·API·타입·라우트·스토어 정리
   - 데이터 흐름(searchParams, API 호출 시점) 파악

4. ✅ **백엔드 API 규약 확인**
   - 기존 백엔드 연동 시: 파라미터 이름(대소문자)·요청/응답 형식 확인

5. ✅ **타입 정의 먼저 작성**
   - 타입은 `api/pages/[module]/[category]/types.ts`에 정의
   - API 응답 구조 명확히

6. ✅ **성능 최적화 필수 적용**
   - 가상 스크롤링
   - GPU 가속
   - watch 최적화

### DON'T (절대 하지 말아야 할 것)

1. ❌ **tomms-lite-front 무시하고 직접 구현**
   - 스타일 불일치 발생
   - 패턴 혼란

2. ❌ **파라미터 대소문자 무시**
   - 백엔드가 대문자 규약이면 연동 실패
   - 데이터 조회 안됨

3. ❌ **ref 경로 잘못 사용**
   - `.value.value` 중복
   - 컴포넌트 통신 실패

4. ❌ **watch 조건 잘못 작성**
   - 빈 값 처리 안됨 (`if (newVal)` 대신 `if (newVal !== undefined)` 등)
   - 폼 필드 동기화 실패

5. ❌ **성능 최적화 생략**
   - 느린 렌더링
   - 사용자 경험 저하
