# 00. 시작하기

## ⛔ 최우선 원칙: 확실하지 않으면 생성 금지

> **모르는 것을 억지로 만들지 마라. 차라리 만들지 마라.**

- 컴포넌트명, import 경로, prop, event, composable이 **이 가이드에 명시되어 있는지** 확인 — 없으면 생성 금지, `// TODO: verify` 주석으로 대체
- API 엔드포인트, HTTP 메서드, 응답 구조가 **가이드에 명시되어 있는지** 확인 — 없으면 생성 금지
- PrimeVue 컴포넌트 사용법이 **가이드에 예시가 있는지** 확인 — 없으면 생성 금지
- **불완전하지만 올바른 코드 > 완전하지만 틀린 코드** — TODO 3개가 fabrication 3개보다 낫다

## 📋 이 문서의 목적

이 가이드는 **Vue 3 + TypeScript (pfy-front)** 프로젝트에서 **신규 화면 개발**과 **기존 화면 수정**을 할 때 따를 **공통 절차와 패턴**을 제공합니다.

- **신규 개발**: 요구사항과 유사 기존 화면을 참고해 `pages`·`api` 구조에 맞춰 구현합니다.
- **수정 개발**: 변경 대상 페이지·컴포넌트·API·타입의 영향 범위를 파악한 뒤 수정합니다.
- **공통**: 본 FrontendGuide의 패턴을 준수합니다.

## 🎯 핵심 목표

1. **신규**: 기존 화면(예: PMDP020, spov010)과 동일한 폴더·컴포넌트·API 패턴으로 구현
   - **PFY(pfy-front)**: 패턴 참고용 샘플은 `src/pages/sy/ds/` 하위(예: `pmdp020`, `spov010`)에 둡니다.
2. **수정**: 기존 코드 스타일·패턴을 유지하며 변경
3. **스타일**: 기존 화면과 동일한 톤앤매너·스타일 가이드 준수
4. **성능 최적화 필수 적용** · **유지보수 가능한 코드 작성**

## 🚫 단일 Vue 파일 금지 (CRITICAL!)

**모든 화면은 반드시 다중 파일 구조로 생성해야 합니다.**

```
❌ 금지: 하나의 vue_page에 SearchForm, DataTable, API, 스타일을 모두 포함
✅ 필수: index.vue + SearchForm + DataTable + API + SCSS 각각 분리
✅ 선택: SumGrid는 스펙에 진행상태별 집계/필터링 요구사항이 있을 때만 생성
```

- `index.vue`는 오케스트레이터 역할만 (ContentHeader + 하위 컴포넌트 import + provide/inject)
- SearchForm, DataTable은 반드시 `components/` 하위에 별도 폴더로 분리
- **SumGrid는 스펙에 진행상태별 집계 필터(예: 저장/의뢰/요청확정 등 상태별 건수 클릭 필터)가 명시된 경우에만 생성** — 단순 목록 조회 화면에는 생성하지 않음
- API 함수는 `api/pages/[module]/[category]/[screenId].ts`에 분리
- 스타일은 컴포넌트별 `.scss` 파일로 분리

## 📛 파일 네이밍 규칙 (CRITICAL!)

### screenId는 반드시 camelCase

**❌ 금지: 모두 소문자**
```
cpmsedupondgedit  ← 잘못됨 (모두 소문자)
cpmsedupondglst   ← 잘못됨 (모두 소문자)
```

**✅ 필수: camelCase**
```
cpmsEduPondgEdit  ← 올바름 (각 단어 첫 글자 대문자)
cpmsEduPondgLst   ← 올바름 (각 단어 첫 글자 대문자)
```

### 네이밍 패턴 정리

| 항목 | 패턴 | 예시 (screenId = `cpmsEduPondgEdit`) |
|------|------|------|
| **화면 폴더** | `[screenId]/` (camelCase) | `cpmsEduPondgEdit/` |
| **페이지 스타일** | `[screenId].scss` | `cpmsEduPondgEdit.scss` |
| **컴포넌트 폴더** | `[screenId]SearchForm/` | `cpmsEduPondgEditSearchForm/` |
| **컴포넌트 .vue** | `[ScreenId]SearchForm.vue` (첫 글자 대문자) | `CpmsEduPondgEditSearchForm.vue` |
| **컴포넌트 .scss** | `[ScreenId]SearchForm.scss` | `CpmsEduPondgEditSearchForm.scss` |
| **DataTable 폴더** | `[screenId]DataTable/` | `cpmsEduPondgEditDataTable/` |
| **DataTable .vue** | `[ScreenId]DataTable.vue` | `CpmsEduPondgEditDataTable.vue` |
| **utils** | `utils/index.ts` (DataTable만) | `cpmsEduPondgEditDataTable/utils/index.ts` |
| **API 파일** | `[screenId].ts` | `cpmsEduPondgEdit.ts` |
| **타입 파일** | `types.ts` (카테고리 공통) | `api/pages/edu/pondg/types.ts` |

**핵심 규칙**:
- 폴더명·파일명에서 screenId 부분은 **반드시 camelCase**
- Vue 컴포넌트 `.vue` 파일명은 **PascalCase** (첫 글자 대문자)
- SCSS 파일명은 Vue 컴포넌트와 동일하게 **PascalCase**
- `index.vue`와 `index.ts`(utils)는 예외적으로 소문자

## 🏗️ 프로젝트 구조 이해

### 1. 프로젝트 위치

모든 Vue 개발은 `pfy-front` 프로젝트에서 진행합니다.

```
pfy-front/            # 🎯 개발 프로젝트
├── src/
│   ├── api/                 # API 레이어 (api/pages/[module]/[category]/)
│   ├── pages/               # 화면 (pages/[module]/[category]/[screenId]/)
│   ├── components/common/   # 공통 컴포넌트
│   └── stores/              # Pinia 스토어
│
pfy/              # Spring Boot 백엔드 (API 연동 시 참고)
```

**개발 흐름**:
- **신규**: 요구사항 정리 → 유사 기존 화면 참고 → `pages/[module]/[category]/[screenId]` 및 `api/pages/...` 구현 → 라우트·API 연동
- **수정**: 변경 대상 `index.vue`·컴포넌트·API·타입·라우트 파악 → 수정 범위 정리 → 구현

## 🔑 핵심 개념

### 1. pfy-front 패턴 준수

**절대 원칙**: pfy-front에 이미 구현된 패턴이 있다면 **무조건 그대로 사용**

```typescript
// ❌ 잘못된 방법 - 직접 구현
const handleProgressClick = (field: string) => {
  // SearchForm의 setValue를 복잡하게 호출
  searchFormRef.value?.searchFormRef?.value?.form?.value?.setFieldValue(...)
}

// ✅ 올바른 방법 - 기존 패턴 그대로 (예: spov010)
const handleProgressClick = (field: string) => {
  // 1. searchParams만 업데이트
  searchParams.value.prg_stat = field;

  // 2. SearchForm의 watch가 자동으로 SelectBox 업데이트
  // (SearchForm 내부에서 watch 구현)
}
```

### 2. searchParams 중심 아키텍처

```typescript
// index.vue (부모)
const searchParams = ref();
provide('searchParams', searchParams);

// SearchForm (자식)
const searchParams = inject('searchParams');
watch(() => searchParams.value?.prg_stat, (newVal) => {
  if (newVal !== undefined && searchFormRef.value?.form) {
    searchFormRef.value.form.setFieldValue('prg_stat', newVal ?? '');
  }
});

// ProgressBar (형제)
const searchParams = inject('searchParams');
const handleClick = () => {
  searchParams.value = { ...searchParams.value, prg_stat: 'S01' };
};
```

**핵심**: `searchParams`를 변경하면 → `watch`가 감지 → 폼 필드 자동 동기화

### 3. API 파라미터 규약

PFY 백엔드는 **MyBatis + camelCase 파라미터**를 사용합니다. 프론트엔드에서 **백엔드 DTO 필드명(camelCase)을 그대로 전달**하면 됩니다. 대문자 변환은 불필요합니다.

```typescript
// camelCase 파라미터를 그대로 전달
searchParams.value = { langCd: 'KO', useYn: 'Y' };

// API 레이어에서 변환 없이 직접 전달 (06_API_백엔드_연동.md 참고)
const result = await selectWindowList(searchParams.value);
```

### 4. Ref 접근 경로 (매우 중요!)

```typescript
// ❌ 잘못된 경로
searchFormRef.value?.form?.value.setFieldValue()

// ✅ 올바른 경로 (Vue의 ref auto-unwrap)
searchFormRef.value?.form.setFieldValue()

// 구조 설명:
// - searchFormRef.value → 화면 SearchForm 컴포넌트 인스턴스
// - .form → SearchForm이 expose한 ref (자동 unwrap됨!)
// - .setFieldValue() → PrimeVue Form 메서드
```

## ⚠️ 시작 전 주의사항

### DO (반드시 해야 할 것)

1. ✅ **pfy-front를 먼저 확인**
   - 유사한 화면이 있는지 찾기
   - 패턴과 스타일 참고

2. ✅ **신규 시: 유사 기존 화면 참고**
   - 같은 모듈/카테고리 내 비슷한 화면의 폴더 구조·SearchForm/DataTable/API 패턴 확인
   - 참고 화면 1~2개를 정한 뒤 그 구조를 따르기

3. ✅ **수정 시: 영향 범위 파악**
   - 변경 대상 페이지·컴포넌트·API·타입·라우트·스토어 정리
   - 데이터 흐름(searchParams, API 호출 시점) 파악

4. ✅ **백엔드 API 규약 확인**
   - 기존 백엔드 연동 시: 파라미터 이름(대소문자)·요청/응답 형식 확인

5. ✅ **타입 정의 먼저 작성**
   - 타입은 `api/pages/[module]/[category]/types.ts`에 정의
   - API 응답 구조 명확히

6. ✅ **성능 최적화 필수 적용**
   - 가상 스크롤링
   - GPU 가속
   - watch 최적화

### DON'T (절대 하지 말아야 할 것)

1. ❌ **단일 Vue 파일로 모든 기능 포함 (CRITICAL!)**
   - SearchForm, DataTable, API를 하나의 vue 파일에 넣지 마세요
   - 반드시 `components/` 하위에 별도 폴더·파일로 분리해야 합니다

8. ❌ **스펙에 없는 SumGrid 자동 생성**
   - SumGrid는 진행상태별 집계 필터가 스펙에 명시된 경우에만 생성
   - 단순 목록 조회/CRUD 화면에서는 SumGrid를 생성하지 마세요

2. ❌ **screenId를 모두 소문자로 작성 (CRITICAL!)**
   - `cpmsedupondgedit` ← 잘못됨
   - `cpmsEduPondgEdit` ← 올바름 (camelCase 필수)

3. ❌ **pfy-front 무시하고 직접 구현**
   - 스타일 불일치 발생
   - 패턴 혼란

4. ❌ **파라미터 대소문자 무시**
   - 백엔드가 대문자 규약이면 연동 실패
   - 데이터 조회 안됨

5. ❌ **ref 경로 잘못 사용**
   - `.value.value` 중복
   - 컴포넌트 통신 실패

6. ❌ **watch 조건 잘못 작성**
   - 빈 값 처리 안됨 (`if (newVal)` 대신 `if (newVal !== undefined)` 등)
   - 폼 필드 동기화 실패

7. ❌ **성능 최적화 생략**
   - 느린 렌더링
   - 사용자 경험 저하
