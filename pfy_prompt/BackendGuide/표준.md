### 4.2 패키지 구조 규칙 (CRITICAL — 절대 위반 금지)

백엔드 패키지는 **모듈 레벨까지만** 사용합니다. 화면코드(screen-level)는 패키지에 포함하지 않습니다.

| 레이어 | 패키지 | 파일 경로 |
|--------|--------|-----------|
| DAO Impl | `biz.{module}.dao` | `src/main/java/biz/{module}/dao/{ClassName}DaoImpl.java` |
| Request DTO | `biz.{module}.dto.request` | `src/main/java/biz/{module}/dto/request/{ClassName}ReqDto.java` |
| Response DTO | `biz.{module}.dto.response` | `src/main/java/biz/{module}/dto/response/{ClassName}ResDto.java` |
| Service Impl | `biz.{module}.service` | `src/main/java/biz/{module}/service/{ClassName}ServiceImpl.java` |
| Mapper XML | — | `src/main/resources/biz/{module}/mybatis/mappers/{ClassName}Mapper.xml` |

**예시 (module=edu):**
```
✅ CORRECT   package biz.edu.dao;
✅ CORRECT   package biz.edu.dto.request;
✅ CORRECT   package biz.edu.service;

❌ WRONG     package biz.edu.cpmseduproglst.dao;     ← 화면코드 삽입 금지
❌ WRONG     package com.biz.edu.dao;                ← com. 접두사 금지
❌ WRONG     package biz.edu.ponl.dao;               ← 화면코드 삽입 금지
```

### 4.3 DTO 표준

> **⚠️ DTO 날짜 필드 타입 규칙 — 반드시 String 사용**
> ReqDto / ResDto의 날짜/일시 필드는 **반드시 `String` 타입**으로 선언해야 합니다.
> `java.util.Date`, `LocalDate`, `LocalDateTime`은 절대 사용 금지입니다.
>
> ```java
> // ✅ CORRECT
> private String educationDate;      // 날짜는 String (예: "2025-01-15")
> private String educationStartDate; // 시작일도 String
> private String educationEndDate;   // 종료일도 String
>
> // ❌ WRONG (컴파일/타입 불일치 에러)
> private Date educationDate;        // Date 금지
> private LocalDate educationDate;   // LocalDate 금지
> private LocalDateTime createdAt;   // LocalDateTime 금지 (AuditBaseDto 제외)
> ```
>
> **이유**: `validateDate(String value, String fieldName)` 헬퍼는 `String, String`을 받습니다.
> `param.setEducationDate(request.getEducationDate())`도 String→String 이어야 합니다.
> DB의 날짜 컬럼은 VARCHAR로 매핑되어 문자열로 처리됩니다.

### 4.3 Service 표준

```java
@Slf4j
@Service
public class SYAR030ServiceImpl {

    @Autowired
    private SYAR030DaoImpl syar030Dao;

    @ServiceId("SYAR030/selectWindowList")
    @ServiceName("화면 목록 조회")
    @Transactional(readOnly = true)
    @PreAuthorize("hasAuthority('ITO_ADM')")
    public List<WindowResDto> selectWindowList(WindowReqDto searchDto) {
        log.debug("Service Method : selectWindowList, Input Param={}", searchDto.toString());
        try {
            return syar030Dao.selectWinList(searchDto);
        } catch (Exception e) {
            throw HscException.systemError("화면 목록 검색 중 오류가 발생했습니다", e);
        }
    }

    @ServiceId("SYAR030/saveWindowList")
    @ServiceName("화면 저장")
    @Transactional
    @PreAuthorize("hasAuthority('ITO_ADM')")
    public void saveWindowList(List<WindowResDto> paramList) {
        log.debug("Service Method : saveWindowList, Input Param={}", paramList.toString());
        List<WindowResDto> insertList = CommonUtils.filterByStatus(paramList, GridStatus.INSERTED);
        List<WindowResDto> updateList = CommonUtils.filterByStatus(paramList, GridStatus.UPDATED);
        List<WindowResDto> deleteList = CommonUtils.filterByStatus(paramList, GridStatus.DELETED);

        for(WindowResDto dto : insertList) {
            if(syar030Dao.checkDuplicateWindowPk(dto)) {
                throw new HscException("CM0001");
            }
            syar030Dao.insertWindow(dto);
        }
        syar030Dao.updateWindow(updateList);
        for(WindowResDto dto : deleteList) {
            syar030Dao.deleteWindow(Arrays.asList(dto));
        }
    }
}
```

#### 필수 어노테이션

| 어노테이션 | 용도 | 예시 |
|------------|------|------|
| `@Service` | 서비스 빈 등록 | `@Service` (파라미터 없이 사용) |
| `@ServiceId("ID/메서드")` | 서비스 식별자 | `@ServiceId("SYAR030/selectWindowList")` |
| `@ServiceName("설명")` | 서비스 이름 | `@ServiceName("화면 목록 조회")` |
| `@PreAuthorize("hasAuthority('권한')")` | 권한 체크 | `@PreAuthorize("hasAuthority('ITO_ADM')")` |
| `@Transactional` | 트랜잭션 처리 | 저장/수정/삭제 시 필수 |

#### ServiceImpl → DaoImpl 메소드 호출 규칙 (CRITICAL)

> **ServiceImpl은 DaoImpl에 실제 존재하는 메소드만 호출해야 합니다.**  
> DaoImpl에 정의되지 않은 메소드를 호출하면 **컴파일 에러**가 발생합니다.

```java
// ✅ CORRECT — DaoImpl에 존재하는 메소드 호출
syar030Dao.selectWinList(searchDto);           // DaoImpl에 selectWinList() 존재
syar030Dao.checkDuplicateWindowPk(dto);        // DaoImpl에 checkDuplicateWindowPk() 존재
syar030Dao.insertWindow(dto);                  // DaoImpl에 insertWindow() 존재

// ❌ WRONG — DaoImpl에 없는 메소드 호출 (컴파일 에러)
syar030Dao.selectDuplicateCount(dto);          // DaoImpl에 이 메소드가 없으면 에러!
syar030Dao.selectUserByUserId(request);        // DaoImpl에 이 메소드가 없으면 에러!
```

**생성 순서:** Mapper XML → DaoImpl → ServiceImpl 순서로 생성하며, ServiceImpl 작성 시 반드시 DaoImpl의 메소드 목록을 확인한 후 호출합니다.

### 4.4 DAO 표준

> **주의**: DaoImpl은 `extends AbstractSqlSessionDaoSupport` **만** 사용합니다.  
> DAO 인터페이스는 생성하지 않으며, `implements XxxDao` 를 **절대 사용하지 않습니다**.

#### 4.4.1 핵심 원칙: DaoImpl은 Mapper XML의 모든 SQL statement에 대응해야 함

DaoImpl은 **기본 CRUD 4개 메소드만 만드는 것이 아닙니다.**  
Mapper XML에 정의된 **모든** `<select>`, `<insert>`, `<update>`, `<delete>` statement에 대해 1:1 대응하는 public 메소드가 있어야 합니다.

> **⚠️ CRITICAL — DaoImpl public 메소드명 절대 규칙**  
> `insert`, `update`, `delete`, `select`, `selectList`, `selectOne` 같은 **bare CRUD 동사만으로 메소드를 선언하면 절대 안 됩니다.**  
> 반드시 도메인 명사(Xxx)를 붙여야 하며, 해당 이름이 Mapper XML의 `<statement id>`와 **정확히 일치**해야 합니다.

```
❌ WRONG — bare CRUD 동사만 사용:
    public int insert(XxxResDto p)       ← 절대 금지
    public int update(XxxResDto p)       ← 절대 금지
    public int delete(List<XxxResDto> p) ← 절대 금지
    public List<?> select(XxxReqDto p)   ← 절대 금지
    public List<?> selectList(XxxReqDto p) ← 절대 금지

✅ CORRECT — Mapper XML statement id와 정확히 일치:
    selectWinList()              ← <select id="selectWinList">
    checkDuplicateWindowPk()     ← <select id="checkDuplicateWindowPk">
    insertWindow()               ← <insert id="insertWindow">
    updateWindow()               ← <update id="updateWindow">
    deleteWindow()               ← <delete id="deleteWindow">
```

> **⚠️ CRITICAL — ServiceImpl의 DAO 호출 규칙**  
> ServiceImpl에서 DaoImpl의 bare CRUD super 메소드(`insert`, `update`, `delete`, `select` 등)를 **직접 호출하면 절대 안 됩니다.**  
> 반드시 DaoImpl에 선언된 **실제 wrapper 메소드명**을 사용해야 합니다.

```
❌ WRONG — ServiceImpl에서 bare 호출:
    xxxDaoImpl.insert(dto)       ← 절대 금지
    xxxDaoImpl.update(list)      ← 절대 금지
    xxxDaoImpl.delete(list)      ← 절대 금지
    xxxDaoImpl.select(dto)       ← 절대 금지
    xxxDaoImpl.selectList(dto)   ← 절대 금지
    xxxDaoImpl.selectOne(dto)    ← 절대 금지

✅ CORRECT — DaoImpl wrapper 메소드 호출:
    xxxDaoImpl.insertWindow(dto)
    xxxDaoImpl.updateWindow(list)
    xxxDaoImpl.deleteWindow(list)
    xxxDaoImpl.selectWinList(searchDto)
```

#### 4.4.2 DaoImpl ↔ Mapper XML ↔ ServiceImpl 관계

```
┌─────────────┐     calls         ┌──────────────────┐    delegates to    ┌─────────────┐
│ ServiceImpl │  ───────────────→  │     DaoImpl      │  ────────────────→ │ Mapper XML  │
│             │                    │                  │                    │             │
│ saveList()  │→ dao.checkDupWinPk() │ checkDupWinPk()│→ super.selectOne() │ <select     │
│             │→ dao.insertWindow()  │ insertWindow()  │→ super.insert()    │  id="..."   │
│             │→ dao.updateWindow()  │ updateWindow()  │→ super.batch...()  │  id="..."   │
│             │→ dao.deleteWindow()  │ deleteWindow()  │→ super.batch...()  │  id="..." > │
└─────────────┘                    └──────────────────┘                    └─────────────┘
```

**규칙:**
- DaoImpl의 메소드명과 Mapper XML의 statement id는 **반드시 일치**해야 합니다
- ServiceImpl은 DaoImpl에 **실제 존재하는 메소드만** 호출해야 합니다
- DaoImpl에 없는 메소드를 ServiceImpl에서 호출하면 **컴파일 에러**가 발생합니다

#### 4.4.3 DaoImpl 메소드 유형별 패턴

| 용도 | 메소드 패턴 | super 위임 | 반환 타입 |
|------|------------|-----------|----------|
| 목록 조회 | `selectXxxList(ReqDto)` | `super.selectList("id", param)` | `List<ResDto>` |
| 단건 조회 | `selectXxx(ReqDto)` | `super.selectOne("id", param)` | `ResDto` |
| PK 중복 체크 | `checkDuplicateXxxPk(ResDto)` | `super.selectOne("id", param)` | `boolean` (count → boolean 변환) |
| 건수 조회 | `selectXxxCount(ReqDto)` | `super.selectOne("id", param)` | `int` |
| 특정 조건 조회 | `selectXxxByYyy(ReqDto)` | `super.selectOne("id", param)` | `ResDto` |
| 등록 | `insertXxx(ResDto)` | `super.insert("id", param)` | `int` |
| 수정 (단건) | `updateXxx(ResDto)` | `super.update("id", param)` | `int` |
| 수정 (배치) | `updateXxx(List<ResDto>)` | `super.batchUpdateReturnSumAffectedRows("id", param)` | `int` |
| 삭제 (배치) | `deleteXxx(List<ResDto>)` | `super.batchUpdateReturnSumAffectedRows("id", param)` | `int` |

```typescript
import { defineStore } from 'pinia';
import { ref } from 'vue';

export const useCommonCodeStore = defineStore('commonCode', () => {
  const dict = ref<Record<string, CommonCodeItem[]>>({});

      loading.value = false;

    }
    return opts;
  }
```

> **⚠️ 중요 — log.error() + throw 중복 금지**  
> `throw HscException.systemError(...)` 또는 `throw new HscException(...)` 직전에 `log.error(...)`를 **함께 쓰지 않습니다.**  
> 예외는 상위 프레임워크에서 이미 로깅되므로 이중 로그가 발생합니다.
>
> ```java
> // ❌ WRONG — 중복 로그
> } catch (Exception e) {
>     log.error("처리 중 오류", e);
>     throw HscException.systemError("처리 중 오류가 발생했습니다", e);
> }
>
> // ✅ CORRECT — throw 만 사용
> } catch (Exception e) {
>     throw HscException.systemError("처리 중 오류가 발생했습니다", e);
> }
> ```

### 10.4 import 자동 정리 (Organize Imports)

코드 생성 후 STS/Eclipse의 **Organize Imports**와 동일한 자동 정리가 적용됩니다:

1. **패키지 교체**: `hone.bom.*` → `aondev.framework.*` 자동 변환
2. **미사용 import 제거**: 코드에서 참조하지 않는 import문 자동 삭제
3. **중복 import 제거**: 동일 클래스의 중복 import 제거
4. **import 정렬**: Java 표준 순서로 자동 정렬
   - `java.*` → `javax.*` → `org.*` → `com.*` → `aondev.*` → `biz.*`
   - 각 그룹 사이에 빈 줄 삽입

```java
// ✅ 정리된 import 예시 (자동 적용)
```
