# HTOMSS Backend 화면 단위 마이그레이션 가이드

## 📋 마이그레이션 개요

**작성일**: 2025-12-04  
**프로젝트**: HTOMSS → tomms-lite Backend 마이그레이션  
**대상**: htomss-olt 업무 모듈 중 선정된 화면

---

## 🎯 마이그레이션 목표

**핵심 목표**: HTOMSS의 선정된 화면(Service)을 tomms-lite-war Backend API로 이관

### 마이그레이션 범위
- **소스**: htomss/htomss-olt (업무 모듈)
- **타겟**: tomms-lite-war (Backend REST API)
- **방식**: 화면(Service) 단위로 독립적으로 이관
- **구조**: TOMMS Portal 스타일 적용

### 진행 방식
- 🎯 **화면 단위 점진적 마이그레이션**
  - 한 번에 하나의 화면(Service) 작업
  - DTO → Service (@ServiceId) → DAO → Mapper 순차 개발
  - @ServiceId 기반 자동 Controller 생성 (수동 작성 불필요)
  - 완료 후 다음 화면 진행

---

## 🔑 핵심 원칙

1. **TOMMS Portal 프로젝트 구조 기준 적용**
   - 패키지 구조: `hsc.tomms.web.{module}.{screen}`
   - 디렉토리 구조: dto/service/dao 분리 (controller 없음)
   - DTO 패턴: Request/Response DTO 분리
   - @ServiceId 어노테이션 기반 자동 Controller 생성
   - @ServiceName 어노테이션으로 메서드 설명

2. **HTOMSS의 DB 연결 및 고유 로직 유지**
   - Oracle ojdbc8 유지 (Tibero 아님)
   - ValidationMeta 패턴 보존
   - 비즈니스 로직 그대로 이관
   - MyBatis Mapper 방식 유지 (HQML → XML)

3. **화면 단위 독립 마이그레이션**
   - 한 번에 하나의 화면(Service)만 작업
   - Backend API 완성 후 테스트
   - 완료 후 다음 화면 진행

4. **@ServiceId 기반 자동 API 생성**
   - Endpoint: `/online/mvcJson/{ServiceId}`
   - @ServiceId 어노테이션으로 API 자동 매핑
   - Request: DTO 객체 (JSON)
   - Response: DTO 객체 (JSON)
   - Controller 수동 작성 불필요

## 🔄 코드 스타일 차이점

### 패키지 구조

| 항목 | HTOMSS | TOMMS Portal |
|------|--------|---------------|
| 최상위 패키지 | `hsc.htomss.web` | `hsc.tomms.web` |
| 업무 패키지 | `hsc.htomss.web.zz.sp` | `hsc.tomms.web.zsample` |
| 마이그레이션 후 | `hsc.htomss.web.zsample` | - |

### 파일 네이밍

| 항목 | HTOMSS | TOMMS Portal |
|------|--------|---------------|
| Service 파일명 | `ZZSP020ServiceImpl` | `ZZSP010ServiceImpl` |
| DAO 파일명 | `ZZSP020Dao` | `ZZSP010Dao` |
| Mapper 파일 | `ZZSP020.hqml` | `ZZSP010Mapper.xml` |

### 데이터 전달 방식

| 항목 | HTOMSS | TOMMS Portal (마이그레이션 목표) |
|------|--------|----------------------------------|
| 입력 데이터 | `Record` (Hone) | `RequestDto` |
| 출력 데이터 | `RecordSet` (Hone) | `ResponseDto` / `List<Dto>` |
| Validation | `ValidationMeta`, `ValidationUtil` | Spring Validation (`@Valid`) + ValidationMeta 병행 |

### Service 어노테이션

**HTOMSS 기존**:
```java
@Service("ZZSP020Service")
public class ZZSP020ServiceImpl {
    public RecordSet selectList(Record input) {
        // ...
    }
}
```

**tomms-lite (TOMMS Portal 스타일)**:
```java
@Service("ZZSP010Service-CRUD 코드스타일 샘플 서비스")
public class FWMybatis001ServiceImpl implements FWMybatis001Service {
    
    @Autowired
    private FWMybatis001Dao dao;
    
    @Override
    @ServiceId("MybatisReadList")  // 이 값으로 API 자동 생성
    @ServiceName("조회-목록")
    public List<UserResDto> selectUserList(UserReqDto input) {
        logger.debug("Input Param={}", input.toString()); 
        return dao.selectUserList(input);
    }
}

// 자동 생성되는 API:
// POST /online/mvcJson/MybatisReadList
// Request Body: UserReqDto (JSON)
// Response: List<UserResDto> (JSON)
```

## ⚠️ 주의사항

1. **화면 단위 독립 작업**
   - 한 번에 하나의 화면(Service)만 작업
   - 여러 화면 동시 작업 금지
   - 완료 및 테스트 후 다음 화면 진행

2. **TOMMS Portal 구조 기준 준수**
   - 패키지: `hsc.tomms.web.{module}.{screen}`
   - 하위 구조: `dto/service/dao` 분리 (controller 폴더 없음)
   - DTO: `dto/request/`, `dto/response/` 명확히 분리
   - @ServiceId 어노테이션 필수 (자동 API 매핑)
   - @ServiceName 어노테이션 추가 (메서드 설명)

3. **비즈니스 로직 보존**
   - HTOMSS ValidationMeta 패턴 유지
   - 원본 Service의 로직 그대로 이관
   - 데이터 타입만 Record → DTO로 변환

4. **MyBatis Mapper 변환**
   - HQML → XML Mapper로 변환
   - 위치: `src/main/resources/.../mybatis/mappers/{Screen}Mapper.xml`
   - Namespace는 DAO interface 풀패키지명과 일치
   - DAO: interface + DaoImpl 클래스 분리
   - 동적 쿼리 태그 MyBatis 문법으로 변환

5. **API 설계 원칙**
   - Endpoint 패턴: `/online/mvcJson/{ServiceId}`
   - @ServiceId 값이 그대로 API 경로가 됨
   - HTTP Method: POST (기본)
   - Request/Response: JSON 형식
   - Controller 코드 작성 불필요
